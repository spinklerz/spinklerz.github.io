High Level Overview of Processes and Memory

Lets go over the workflow of a process. If we have a python script named HelloWorld.pythat has the code 

print("Hello, World")
and we ran it with python3 HelloWorld.py. The operating system would then create a new process, as might you imagine there are many processes running on a computer. There are many different algorithm's that determine when each process gets executed. For simplicity lets say our algorithm operates on queue( FIFO ) and the process is named p1. When our process gets popped for execution there are 3 stages a CPU goes through to process p1. The CPU receives p1 as a batch instruction, it fetches the instruction, decodes it, then executes it. Its very important to note, these three stages are happening concurrently meaning there is an instruction at each phase in the process at the same time. Before a process is executed it is assigned something call virtual memory. You can think of virtual memory as fake memory, memory that hasn't been allocated yet. When a process is ready to execute there is a data structure called a page table that translates virtual memory into actual memory. Virtual memory is usually many times larger then the actual space available in memory. Memory works like a cache where it keeps commonly used address spaces and those who arent used that much get pushed to the disk. A common algorithm is Least Frequently Used (LFU). Where is ASLR in all this?

ASLR

Address Space Layout Randomization is what it sounds like, the randomization of address spaces. When a process is executed it is given a range called ASLR window. Disk and memory are usually partitioned in things called pages. Which are just slices of memory usually around 4KB. Now for example lets say our ASLR window is around 512 we multiply that by the page size 4KB and get 512*4KB = 2MB. So the process can start anywhere within that 2MB range. The important part is the stack can only start at 512 positions. Which makes injection very predicatble. Here is how this likely went:

  • Attacker did some static analysis on source code, and found a vulnerability
  • Without ASLR the attacker predicted or mapped out how far away from the stack the address location of the vulnerability is.
  • In our example there are 512 position the stack could start, so they attacker would guess and injection the malicous payload at n locations of the window

How an attacker would typically perform this attack is by examining the code, identifying potential memory leaks or buffer overflows, and then by understanding the address space positions they can inject malicous to redirect the program flow back to there mailous script/program etc...

Thats a basic run through but lets move onto what happened with Ivanti

Ivanti

After reading the PoC provided by watchTowr(Check it out: Link) it is actually quite simple. The attackers chained two vulnerabilities together, one that gave unauthed directory traversal to sensitive materials, and the other that led to a RCE. Analyzing the PoC it was just a GET request with an encoded java payload to the format paramter. To save you guys time here is the decoded payload 

format=${"".getClass().forName('java.lang.Runtime').getMethod('getRuntime').invoke(null).exec('id')}'
It's very interesting to learn new things even if it outside a domain that I currently know. This taught me a new possible indicators of compromise, which could be when a system crashes because often times injecting code into ASLR memory locations if successful or failure will cause the program to crash.

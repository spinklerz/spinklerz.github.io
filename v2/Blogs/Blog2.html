Introduction

I've been tracking a subdomain takeover that I found via subdomain hijacking. How this happened is that I saw odd activity being indexed by Google on a retail company's subdomain. So I decided to check the DNS records and saw a dangling CNAME that pointed externally. The dangling CNAME led me through a trail across this criminal's infrastructure and inner workings. Just to preface, I reported the site to CISA and to the companies' SOCs. As of right now, the CNAME connection has not been severed, so it is still vulnerable to my knowledge. Although this connection is still intact, I believe the malicious actors have "moved on" from that exposed CNAME domain. Also, I've taken the liberty to redact any information to protect myself as well as the currently exposed retail store.

The Hunt

It all started on Google looking for hiking boots, until I found weird search results from a popular retail store. The indexed result on Google was a subdomain that contained results that described movies, books, and quizzes. Using public sources like dnsdumpster, dig, nslookup, and some other online DNS recon tools most commonly used, I wanted more information. Let's say the legitimate subdomain subdomain.legit_retailer.com and its externally pointed CNAME is subdomain.malicious1.com I decided to check out malicious1.com and found out some interesting stuff. What jumped out to me was the fact the subdomain server was held in Spain while having the name servers in the United States. The SPF records contained domains that were flagged by VirusTotal and typosquats. I believe the subdomain and origin were hosted on a web hosting service due to the fact the default host name resolved as malicious_domain.ruttearful.store. I tried to resolve the server as it is accepting HTTP and HTTPS, but I get sent to auto:blank. Then I found out that the CN provided was named malicious_site_2.com. Malicious site 2 was actually resolvable! At first it looked like a legitimate Chinese website, although none of its webpages worked. And the API calls were weird, to say the least.... But anyways, it was obviously a front. What did this website do?

South Africa

Now what was interesting about this site is that it originally was hosted in South Africa on a seemingly disposable 1.8.0 nginx server. As of today, that has changed, and they are now based in San Jose as part of an ISP and are using their cloud services as of today to host a website that uses masked coding techniques to hide commands like document.write() and iframe injections to evade browser detection to deliver malware and adware to the user. The code is quite simple; the script basically queries what I'm pretty sure is the C2 and injects the response from the C2 into the webpage.

Updates

So malicious_domain2 landed in San Jose. Amazingly enough, I reported it to the ISP that owned that IP range, and they were able to take that domain down! If you're wondering why I went to the ISP, it's because the hosting provider was not made obvious via DNS records.

Now this is where things get more interesting. From the San Jose-based server/VPS/etc., they moved back to South Africa. But here I noticed one of the malicious PHP scripts they included _paq.push(["setCookieDomain", *.malicious_domain3_dga.com. Now I'm not exactly sure what this does, but if I had to guess, it looks like it attempts to fingerprint the user or set a cookie using this malicious domain. Interestingly enough, this new domain was a replica of malicious_domain2. But what I noticed is the constant direct references/calls to malicious_domain4. Now upon looking at malicious domain 4, nothing stands out, and originally I thought nothing of it as it seemed like a legitimate analytics login page. If we dive deeper, the login page is dysfunctional; the password reset feature literally only gives the option to enter a username and then create a new password. And at the top is a logo for a web service called Matomo. Matomo is a web-based open-source analytics platform. As of now malicious_domain3, 4 and 2 are still functional. In domains 3 and 4, what happens is they call malicious domain 4 to get malware/deceptive content/etc, to then inject the response to overwrite the webpage content.

Learning

It feels weird to say, but I really should thank this group for making it so easy to track them. It's crazy how many times you can pivot and still use the same domain name, thinking you won't be tracked. Anyway, they have really taught me a lot. Even the DNS history is very interesting in that it's common to pivot and change name servers frequently. Also, some things that didn't make sense in the SOA records were that many times the refresh and retry time would be longer than the expire time. What I believe happens in this case is when a DNS query fails, the secondary DNS server stops serving that record so that it is only available on the master server. Which is odd to me; maybe it was just caught in the DNS history that way.

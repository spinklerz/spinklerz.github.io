Introduction

I've been tracking a subdomain takeover that I found via subdomain hijacking. How this happened is I saw odd activity being indexed by google on a companies subdomain. So I decided to check the dns records and saw a dangling cname that pointed externally. I reported the site to CISA and the company SOC. As of right now the cname connection has not been severed so it is still vulnerable to my knowledge.

The Hunt

Now using dnsdumpster, dig, nslookup and some other online dns recon tools(most commonly used) I wanted more information. Let's say the malicous subdomain was agent.special.com I decided to check out special.com and found out some interesting stuff. What jumped out to me was the fact the subdomain server was held in Spain while having the name servers in the United States. I believe subdomain and origin were hosted on a web hosting service due to the fact the default host name resolved as malicous_domain.ruttearful.store Now the server is accpeting http and https but I get sent to auto:blank. Then I found out that the cn provided was named malicous_site_2.com. Malicous site 2 was actually resolvable at first and looked like a legitmate chinese website although none of its webpages worked and the api calls were weird to say the least.... But anyways it was obviously a front.

South Africa

Now what was interesting about this site is that originally maybe like a couple week ago ( end May ) the server was hosted in South Africa on a seemingly disposable 1.8.0 nginx server. As of today that has changed and are now based on San Jose apart of PEG-SV and are using there cloud services as of today to host a chinese website that uses masked coding techniques to hide commands like document.write() and iframe injection to evade browser detection to deilver malware and adware to the user.

Learning

It feels weird to say but I really should thank this group for making it so easy to track them. Its crazy how many times you can pivot and still use the same domain name thinking you wont be tracked. Anyways they have really taught me a lot even the dns history is very interesting in that its common to pivot and change name servers frequently. Also some things that didn't make sense in the SOA records many times the refresh and retry time would be longer then the expire time. What I believes happens in this case is when dns query fails it automatically fails and the secondary dns server stops serving that record. Which is odd to me, maybe it was just caught in the dns history that way.
